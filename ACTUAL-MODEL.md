# Emergence London - Actual Business Model

## What It Actually Is

**An event aggregator with a headline brand** - not an event organizer (yet).

**Core Function:**
- Aggregate existing events happening April 28-30, 2026 across London's tech scene
- Create unified discovery layer (map + calendar)
- Brand it as "Emergence London" to create cohesion and narrative
- NOT directly organizing/paying for events (hosts run their own stuff)
- NOT providing "protocol" infrastructure (at least not yet)

**Think:**
- Product Hunt for London tech events
- A festival brand that aggregates existing programming
- Discovery platform + sponsorship vehicle

**NOT:**
- A conference we're organizing
- Paying hosts to create programming
- Building actual technical infrastructure for event management

---

## Revenue Model

**Primary:** Attract sponsors who want brand association with "London's tech scene, emerging"

**Sponsor funding goes to:**
1. **Closing party** (April 30) - the ONE event we actually organize
2. **Featured/editorial content** - highlight specific events on the platform
3. **Platform/infrastructure costs** - map, site, aggregation
4. **Future programming** - if this works, we can organize more ourselves later

**NOT:** Direct payment to individual hosts/nodes

---

## Current Stage: Pre-Launch (Signal Collection)

**Goal:** Gauge interest before building

**What we need:**
1. **Email signups** - measure demand, build audience
2. **Potential sponsors** - companies interested in associating with the brand
3. **Host commitments** - venues/organizers willing to list their events on the platform
4. **Proof of concept** - can we get 20-30 events aggregated for April 28-30?

**What we're NOT doing yet:**
- Building the actual Luma integration
- Creating the full event map
- Formal sponsorship packages
- Organizing programming ourselves (except the closing party)

---

## User Journeys (What People Actually Do)

### Attendees
1. Discover Emergence brand/site
2. Browse aggregated events on map/calendar (when live)
3. RSVP directly through event hosts (Luma, Eventbrite, host sites)
4. Navigate between events across London
5. Attend closing party (if we have one)

**Value prop:** One place to discover everything happening in London tech that weekend

### Hosts
1. Discover Emergence platform
2. Submit their existing event to be included
3. Get free visibility/discovery through the aggregation
4. Maybe get featured/editorial if their event fits the narrative
5. Run their event themselves (no support from us)

**Value prop:** Free marketing/discovery for their event

### Sponsors
1. Associate brand with "London's emerging tech scene" narrative
2. Funding goes toward closing party + platform + featured content
3. Brand placement on site, at closing party, in featured content
4. NOT directly funding individual events (unless they want to separately)

**Value prop:** Align with aspirational brand positioning London as innovative hub

## What The Site Actually Needs To Do

### Primary Functions

1. **Capture interest signals**
   - Email signups from potential attendees
   - Host event submissions
   - Sponsor inquiries
   - Measure: # signups, quality of submissions

2. **Communicate the value prop clearly**
   - For attendees: "Discover everything happening in London tech, April 28-30"
   - For hosts: "Get free visibility for your event"
   - For sponsors: "Associate your brand with London's tech momentum"

3. **Build narrative/brand**
   - Position London as an emerging tech hub
   - Frame distributed events as more interesting than single conferences
   - Create FOMO / sense of moment

4. **Establish credibility**
   - Show the density exists (300+ companies on map)
   - Reference real venues/areas (Shoreditch, King's Cross)
   - Make it feel inevitable/real even though it's not built yet

### Secondary Functions

1. **Qualify leads**
   - Which hosts have good events worth featuring?
   - Which sponsors are serious?
   - Which attendees are actually in London/relevant?

2. **Set expectations**
   - Map launches April 1 (buy time to build)
   - Events are hosted independently (manage expectations)
   - This is discovery, not ticketing (clarify model)

## What Success Looks Like (Pre-Launch)

**Good signals:**
- 500+ email signups before April
- 10+ host event submissions that are genuinely good
- 2-3 sponsor inquiries with real budgets
- Proof we can aggregate 20-30 events for the dates

**Bad signals:**
- Just randoms signing up with no intent
- No host submissions (means the value prop doesn't work)
- No sponsor interest (means the brand doesn't resonate)

## Mental Model Alignment

### What we're ACTUALLY selling:

**To Attendees:** Discovery/curation (make London's scene visible)
**To Hosts:** Free marketing/visibility
**To Sponsors:** Brand association + community access

### Psychological levers we should use:

1. **FOMO / Scarcity** - "One weekend. Fifty venues. Miss it and you're not part of the moment."
2. **Social Proof** - "250+ companies. 12 unicorns. £2.3B raised. The network is real."
3. **Unity Principle** - "You're already part of London's tech scene. Now make yourself visible."
4. **Reciprocity** - Give free discovery/aggregation, hosts/attendees reciprocate with participation
5. **Bandwagon Effect** - Frame it as inevitable momentum, not a risky experiment
6. **Contrast Effect** - "Traditional conferences put you in one room. Emergence gives you the whole city."

### What we should STOP implying:

1. **We're organizing everything** - No. We're aggregating.
2. **Hosts get paid** - No. They get visibility.
3. **There's technical infrastructure** - Not yet. Just discovery/aggregation.
4. **This is fully built** - No. We're testing demand first.

## Honest Copy Audit

### Current problems:

**"We provide the protocol and route the network to your door"**
- Reality: We're just listing events on a map. We're not routing anything.
- Fix: "We make your event discoverable to thousands exploring London's tech scene."

**"Each node hosts autonomous programming... discoverable through the central protocol"**
- Reality: Events already exist. We're just aggregating them.
- Fix: "Events run independently. We make them all discoverable in one place."

**"Sponsors fund the ecosystem"**
- Reality: Sponsors fund our closing party + platform costs.
- Fix: "Sponsors support the platform and closing party, making discovery free for everyone."

**Stats section feels premature**
- "50+ Active Venues" - Do we have 50 commitments? If not, this is dishonest.
- Fix: Use London ecosystem stats (real), not Emergence stats (aspirational).

## Recommended Site Structure

### Home Page
- **Hero:** Clear value prop for all three audiences
- **Problem:** London's tech scene is fragmented, hard to discover
- **Solution:** One weekend, one map, all the events
- **Social Proof:** Real London ecosystem stats (not fake Emergence metrics)
- **CTA:** Three clear paths (Attend / Host / Sponsor)

### /events
- **Primary:** "Events aggregated from across London's tech scene"
- **Timeline:** Map launches April 1, full event list April 1
- **CTA:** Sign up for early access

### /host
- **Value Prop:** Free visibility for your event
- **Process:** Submit your event → We review → You get listed → You run it yourself
- **CTA:** Submit event (when ready) or register interest (now)

### /sponsor
- **Value Prop:** Associate with London's tech momentum
- **What funding supports:** Closing party, platform, featured content
- **NOT:** Direct funding of individual events (unless clarified separately)
- **CTA:** Get sponsor deck / inquiry

### /map
- **Show the density** - LondonMaxxing embed proves the ecosystem exists
- **Frame:** "This is what makes Emergence possible"
- **CTA:** Sign up to see event layer when it launches

### /about
- **Honest positioning:** We're aggregating London's existing momentum, not creating it
- **Vision:** Make the scene visible to itself
- **Team:** (when ready)
